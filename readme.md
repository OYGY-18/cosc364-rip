## Seamless Integration Between Components

All modules in the project are designed to work together, and for the most part they integrate smoothly. The overall architecture separates concerns well: **configuration parsing** (`Config` class), **validation** (`PortValidator`), and the **RIP protocol logic** (`RIP` class and its helpers) are in distinct components. It instantiates a `Config` object with the given file path and calls `config.parse_config()` to load and validate the settings ([main.py](file://file-7PRzKyR617v8USPhnbdaj1#:~:text=,Router%20%7Bconfig.router_id%7D%3A%20Configuration%20loaded%20successfully)). If any required parameter is missing or invalid, `Config.parse_config` will handle the error and exit the program, so by the time main continues, we know we have a valid configuration. This design ensures that only correct data is passed forward into the routing process.

- **Validation Utility:** The `PortValidator` class is used by `Config` to check port number values. This is a good modular design choice – it keeps the parsing code clean by delegating range checks and duplicate detection to a dedicated utility. For example, `Config.parse_config()` uses `PortValidator.validate_port` for each port string ([config.py](file://file-3EDrcPzEFFVmk7Sh8sUMv3#:~:text=,port%20and%20add%20to%20list)) ([config.py](file://file-3EDrcPzEFFVmk7Sh8sUMv3#:~:text=,if%20neighbor_id%20in%20self.neighbors)), and calls the `check_duplicate_ports` and `check_input_output_conflict` static methods to enforce uniqueness ([config.py](file://file-3EDrcPzEFFVmk7Sh8sUMv3#:~:text=,except%20KeyError%20as%20e)). - **Main to RIP Initialization:** After parsing, the main function creates a `RIP` instance: `rip = RIP(config.router_id, config.input_ports, config.neighbors)` ([main.py](file://file-7PRzKyR617v8USPhnbdaj1#:~:text=,Listening%20for%20messages)). The `RIP` class’s constructor is defined to accept the list of output neighbors (with their port, cost, and ID) as the third argument, but here `config.neighbors` (a dictionary) is passed instead of `config.outputs` (the list of tuples). Since `config.outputs` is a list of `(port, cost, neighbor_id)` tuples, it will match what `RIP.__init__` expects. Alternatively, the `RIP` class could be refactored to accept the neighbors dict directly, but using the outputs list is simpler given how the code is currently written. Aside from this bug, the data flow from config to RIP is logically correct (the config provides all the info RIP needs: the router’s own ID, the input ports to bind, and the neighbors list).

If an input port is out of range or not a number, it will say “Invalid value in config file: ...” with details. These messages make it easy to troubleshoot config issues. The program will not proceed to run with an invalid configuration, which is good for preventing undefined behavior later. This robustness in config handling meets the assignment expectation that the program should perform basic syntax and consistency checks ([Assignment.pdf](file://file-89EFgjoDKfmuLNaXZvtygb#:~:text=ish%20or%20any%20other%20similarly,set%20at%20least%20the%20following)).

- **Socket Binding and Networking:** When creating UDP sockets for the input ports, the code handles errors gracefully. It tries to bind each specified port on localhost, and if any bind fails (e.g., the port is already in use by another process or instance), it catches the exception and prints an error like “Failed to bind socket on port X: [error details]” ([config.py](file://file-3EDrcPzEFFVmk7Sh8sUMv3#:~:text=for%20port%20in%20self,e)), then exits. This prevents the program from continuing in a half-functional state. By binding to `"localhost"`, the routers are effectively isolated to the local machine, which is appropriate for the assignment’s design of simulating multiple routers on one host. The use of `SO_REUSEADDR` on the sockets in `RIP.create_sockets()` helps in quickly restarting the program in case a socket was left in a TIME_WAIT state or similar, although typically for UDP on localhost this isn’t a big issue. The networking code (send and receive) assumes a reliable local environment (no packet loss or corruption, which is fine for local socket simulation). If a send fails for some reason, it prints a message but continues operation for other neighbors ([rip.py](file://file-XRd9XHRUdvn9cVKZPAmLxv#:~:text=sock%20%3D%20socket,finally%3A%20sock.close)). The receive loop uses `select` with a timeout of 1 second, meaning the router wakes up at least every second to check for incoming data. This ensures it doesn’t block forever and can also check its timers regularly. Overall, the network I/O functionality is solid and handles errors where expected.

- **Processing Incoming Packets:** The `RIP.receive_rip_messages()` loop continuously listens on all input sockets and processes packets one by one ([rip.py](file://file-XRd9XHRUdvn9cVKZPAmLxv#:~:text=def%20receive_rip_messages%28self%29%3A%20,safe%20manner)). It is robust in that it catches any invalid packet (through `RIPPacket.unpack()` returning `None`) and simply ignores it ([rip.py](file://file-XRd9XHRUdvn9cVKZPAmLxv#:~:text=packet%20%3D%20RIPPacket,command%20%3D%3D%202)). Well-formed RIP packets with command 2 (response) trigger the routing table update logic. The update algorithm in `process_rip_message` is a correct implementation of the Distance Vector update rule:
  - It identifies the neighbor (sender) by the router_id field in the packet.
  - If the sender is not recognized as a configured neighbor, the packet is ignored ([rip.py](file://file-XRd9XHRUdvn9cVKZPAmLxv#:~:text=neighbor_id%20%3D%20packet,neighbors%3A%20return%20route_changed%20%3D%20False)) (this avoids poisoning the table with routes from non-neighbors).
  - For each route entry in the incoming packet, it calculates `new_cost = cost_to_neighbor + neighbor_metric`. The `cost_to_neighbor` is this router’s cost to reach the sending neighbor (from `self.link_costs`), and `neighbor_metric` is the metric the neighbor has for that destination (capped at 16 for infinity) ([rip.py](file://file-XRd9XHRUdvn9cVKZPAmLxv#:~:text=for%20dest_id%2C%20metric%2C%20next_hop%20in,routing_table)). This effectively adds the link cost to the neighbor’s reported distance, which is the core Bellman-Ford update step.
  - It then compares this `new_cost` to its current known cost for that destination:
    - If the destination is itself, it skips (a router never updates a route to itself from neighbors) ([rip.py](file://file-XRd9XHRUdvn9cVKZPAmLxv#:~:text=route_changed%20%3D%20False%20for%20dest_id%2C,cost%20to%20dest%20via%20neighbor)).
    - If the destination is not in its table and the new route is reachable (cost ≤ 15), it adds the route via this neighbor ([rip.py](file://file-XRd9XHRUdvn9cVKZPAmLxv#:~:text=route_changed%20%3D%20True%20else%3A%20if,dest_id%5D%20%3D%20RIPEntry%28dest_id%2C%20new_cost%2C%20neighbor_id)).
    - If the destination is already in the table:
      - If the next hop for that destination *is* this neighbor, it updates the metric to the new_cost (this covers the case where the route was previously going through this neighbor and perhaps the neighbor’s metric changed) ([rip.py](file://file-XRd9XHRUdvn9cVKZPAmLxv#:~:text=if%20dest_id%20in%20self,time)). If the metric changed, it flags that a route_changed happened. It also always refreshes the timer for routes learned from this neighbor whenever an update is received, even if the metric didn’t change (preventing unnecessary timeouts) ([rip.py](file://file-XRd9XHRUdvn9cVKZPAmLxv#:~:text=if%20entry,time%28%29%20else)).
      - If the next hop for that destination is different (some other neighbor), then it checks if this new route is better (lower cost) or if the current route is marked unreachable (metric 16). If so, it switches to use this neighbor’s route (a better path found) ([rip.py](file://file-XRd9XHRUdvn9cVKZPAmLxv#:~:text=else%3A%20if%20new_cost%20,dest_id%5D%20%3D%20RIPEntry%28dest_id)). This is how the routing table converges to the shortest paths — the code effectively implements the “if (D < current_D) then update” logic.
    - If a route is present and the neighbor’s route is worse (higher cost) and that neighbor isn’t the current next hop, the code does nothing (which is correct – it ignores inferior routes).
  - If any route was added or updated in the table, it sets a `route_changed` flag and after processing all entries, if `route_changed` is True, it triggers an immediate `send_rip_update()` ([rip.py](file://file-XRd9XHRUdvn9cVKZPAmLxv#:~:text=new_cost%2C%20neighbor_id,print_routing_table)). As noted earlier, this is a slightly aggressive triggered update strategy (it will trigger on any improvement), but it does ensure fast propagation of changes. Functionally, it means the routing information will quickly reach all routers, which is actually robust in terms of convergence speed (just not minimal in message count).
  - After processing a RIP response, the code prints the new routing table state ([rip.py](file://file-XRd9XHRUdvn9cVKZPAmLxv#:~:text=if%20route_changed%3A%20self.send_rip_update%28%29%20,print_routing_table)). This provides immediate feedback on what changed due to the incoming update.

  This logic covers all typical cases for distance vector routing. It prevents loops primarily via the split horizon mechanism and by ignoring routes from non-neighbors. One scenario to consider: what if two routers have equal-cost paths and one update arrives slightly later? The algorithm as written will simply accept the first better path it sees. If another path of equal cost comes later, it won’t replace the current one (since new_cost < entry.metric will be false if they are equal and entry.metric is not 16). RIP doesn’t typically do load balancing in this project scope, so that’s fine. The implementation chooses one shortest path and sticks with it unless a strictly better one appears or the current one fails, which is standard.

- **Periodic Updates and Timeout Handling:** The periodic update thread calls `send_rip_update()` every 30 seconds and also calls `check_routing_table()` on the same cycle ([rip.py](file://file-XRd9XHRUdvn9cVKZPAmLxv#:~:text=while%20True%3A%20time.sleep%28update_interval%29%20,print_routing_table)). This design means that every 30 seconds, the router not only sends an update to neighbors but also checks for any routes that have expired (180s with no refresh). If a route has exceeded the timeout, the code marks it as unreachable (metric 16) and prints a message “Route to X timed out. Marking as unreachable.” ([rip.py](file://file-XRd9XHRUdvn9cVKZPAmLxv#:~:text=current_time%20%3D%20time,timer%20%3D%20current_time)) ([rip.py](file://file-XRd9XHRUdvn9cVKZPAmLxv#:~:text=,dest_id%7D%20after%20garbage)). It then triggers an update immediately (inside `check_routing_table` it calls `send_rip_update()` when a route times out) to notify neighbors of the change. That triggered update happens within the locked section of the periodic thread, ensuring no race with the main thread. If a route was already in an “unreachable” state (metric 16) and 120 more seconds pass, it gets removed entirely from the table and a message “Removed route to X after garbage collection.” is printed ([rip.py](file://file-XRd9XHRUdvn9cVKZPAmLxv#:~:text=unreachable.%22%29%20else%3A%20if%20current_time%20,dest_id%7D%20after%20garbage%20collection)). This garbage collection prevents the table from growing stale entries indefinitely. The combination of periodic refresh and timeout checking is robust: even if no new updates arrive, the periodic thread will eventually detect failures. And if many updates are happening, triggered updates will propagate changes faster than waiting for the next period.

- **Concurrency and Synchronization:** The use of a threading Lock around routing table updates (in both `process_rip_message` and the periodic update) is a critical robustness feature. Without it, a periodic update could overlap with processing an incoming message and potentially cause inconsistent reads/writes of the routing table. By locking, the code ensures only one thread alters or reads the table at a time. This is important because Python’s threads do run concurrently on I/O waits (even though the GIL prevents true parallel execution, the lock is still needed to coordinate logical concurrency). The fact that the lock is acquired for a relatively short time (just to update routes and possibly send out updates) means it’s unlikely to be a performance bottleneck, and the select loop’s 1-second tick ensures the periodic thread won’t be starved. This design appears deadlock-free and safe.

- **Edge Cases and Fault Tolerance:** Several edge conditions are handled:
  - A router cannot list itself as a neighbor (caught in config), avoiding a trivial loop or self-route error ([config.py](file://file-3EDrcPzEFFVmk7Sh8sUMv3#:~:text=cost%20%3D%20self,neighbor_id%7D%20in%20outputs)).
  - Duplicate neighbor definitions are caught (so the routing logic won’t process the same neighbor twice with conflicting info) ([config.py](file://file-3EDrcPzEFFVmk7Sh8sUMv3#:~:text=itself%20as%20a%20neighbor,neighbor_id%5D%20%3D%20port)).
  - If a router had no neighbors (outputs list empty), the config parser would currently treat that as an error (“Outputs list is empty”). In a real scenario, a standalone router with no neighbors is not particularly useful, but if needed, one could allow an empty outputs list. The code as given will force at least one neighbor. This is an acceptable assumption given the assignment’s context of networks – it just means every router in the test network should have at least one link.
  - If multiple input ports are configured (meaning the router has multiple neighbors), the select loop handles them concurrently. The router will process packets from all neighbors, and the logic naturally allows routes to propagate across multiple hops. There is no hard-coded limit to the number of neighbors; it should work for “arbitrarily many” input ports as long as they are listed in one line in the config (which matches the spec allowing arbitrarily many input ports) ([Assignment.pdf](file://file-89EFgjoDKfmuLNaXZvtygb#:~:text=%E2%80%93%20You%20can%20specify%20arbitrarily,but%20all%20in%20one%20line)).
  - The code doesn’t implement split horizon for request messages (since requests aren’t used), and it doesn’t implement any hold-down mechanisms – these are not required by the assignment (those would be advanced features in RFC Section 3 or 4 that we’re not doing). The absence of hold-down is not a problem; in fact, the assignment simplified triggered updates to only triggered on invalidation, which the code slightly overshoots by triggering on any change, but no hold-down means if a better route comes along immediately after a failure, the router will adopt it (which is fine for our scenario).
  - The signal handler ensures that even if the user stops the program with Ctrl+C, it closes sockets cleanly. If the program is killed by other means, sockets will close when the process dies anyway. Since each router is a separate process in this design, shutting down one doesn’t directly affect the others except that the others will notice the lack of updates and time out that router’s routes.

- **Testing Scenarios:** Based on this implementation, we can expect the following behavior in a test:
